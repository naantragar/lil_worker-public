# markdown-new — Policy

## Использовать ДА

- Публичные страницы: статьи, документация, README, блог, вики, changelog, пресс-релизы
- Нужно извлечь текст/структуру, сделать конспект, ответить по содержимому
- HTML шумный (меню, боковые панели, скрипты) — Markdown чище и короче
- Страницы с таблицами/списками — структура сохраняется лучше

## Использовать НЕТ (категорически)

- Приватные/внутренние ресурсы: intranet, админки, панели провайдеров, роутеры,
  Jenkins/GitLab internal, Grafana, всё что требует VPN или внутренней сети
- URL с секретами: ?token=, ?auth=, ?key=, magic-link, reset password, invite links,
  S3 signed URL и аналоги
- Личные данные: документы, счета, медданные, личные кабинеты, переписки
- Локальные адреса: localhost, 127.x, 10.x, 172.16–31.x, 192.168.x, .local, raw-IP
- Не-HTTPS схемы
- Когда нужен 100% дословный текст (юридические тексты, точные формулировки, условия оферт)

## Правила "если сомневаешься"

1. **Публичности**: нельзя открыть в инкогнито без логина/куки/VPN → не использовать
2. **Секретов**: в URL есть намёк на токен/ключ/подпись → не использовать
3. **Чувствительности**: содержимое не хотел бы пересылать третьей стороне → не использовать
4. **Точности**: нужен дословный текст → вручную / официальный PDF / первоисточник

## Алгоритм выбора режима

- По умолчанию: `--method auto` (дешевле всего, обычно достаточно)
- `--method browser`: только если страница пустая без JS или SPA-приложение
- `--retain-images false`: для конспектов и смысловых выжимок
- `--retain-images true`: если картинки важны (инструкции, схемы, примеры)
